# ADR 0071 — in-process provenance verification for `wbhk upgrade` (DIST-7 follow-up)

- status: accepted (the deferred in-CLI signature verification from ADR-0069).
- date: 2026-06-25
- scope: new `packages/cli/src/provenance.ts` (+ test) + `src/trusted-root.json` (embedded) +
  `scripts/gen-trusted-root.mjs`; a `verifyBinaryProvenance` io seam (`io.ts` + `context.ts`); `wbhk upgrade`
  integration + a `--no-verify-provenance` flag; the `@sigstore/{verify,bundle,protobuf-specs}` deps;
  npm `engines` bumped to node >= 22.
- relates: ADR-0069 (the attestation this verifies — it deferred the in-CLI check), ADR-0067 (`wbhk upgrade`).
- review severity: high (security-critical verification). `/code-review` + `/security-review`.

## context

ADR-0069 added sigstore/SLSA build provenance to the binaries but deferred in-CLI verification (`wbhk upgrade`
gated on the checksum only). The founder asked for real in-process verification so an upgrade proves the new
binary was built by this repo's CI — closing the gap where a fully-compromised release (tampered checksums.txt
too) could be installed.

## decision

1. **`wbhk upgrade` verifies the binary's provenance in-process, strict by default.** After the checksum, it
   fetches the attestation from GitHub's **public** attestations API (`/repos/.../attestations/sha256:<digest>`,
   no auth) and verifies it with sigstore-js **before** replacing the binary. `--no-verify-provenance` is the
   escape hatch (air-gapped / sigstore outage → checksum-only).

2. **What's verified (and the load-bearing extra check).** The verifier proves the in-toto SLSA statement was
   signed by a Fulcio cert whose identity is **pinned**: SAN `= release-cli.yml@refs/tags/cli-v*` (an anchored
   regex) + OIDC issuer `= token.actions.githubusercontent.com`, with Rekor inclusion + an SCT required.
   Sigstore proves the statement is *authentically signed*, **not that it's about our binary** — so we
   separately confirm the statement's `subject.digest.sha256` equals the downloaded binary's. Omitting that
   would let a valid attestation for a *different* artifact pass; it's unit-tested.

3. **Embedded trust root, not TUF.** We verify against the sigstore public-good trusted root **embedded** at
   build (`src/trusted-root.json`, regenerated by `gen-trusted-root.mjs` if it rotates — rare). `@sigstore/tuf`
   fetches the root over the network and bootstraps from a `seeds.json` that **`bun --compile` drops** (→
   "root signed by 0/3 keys"); embedding removes the TUF/network dependency and works in the compiled binary.

4. **A bun-only `crypto.verify` shim.** bun's `crypto.verify` **throws on an `undefined` algorithm** for
   ECDSA, whereas Node derives the digest from the EC curve; sigstore-js calls `verify(undefined, …)` for
   every EC check (Rekor SET, cert chain, SCT), so all verification failed under the bun binary ("inclusion
   promise could not be verified"). A bun-only, idempotent shim supplies the curve-matched digest when none is
   given — pass-through for any explicit algorithm, so it can't weaken a defined-digest verification. With it,
   the FULL verification (incl. transparency-log inclusion) passes under bun.

5. **node >= 22.** The bundled sigstore verifier requires it; node 20 is EOL. The bun binary embeds its own
   runtime, so only the npm package's `engines` is affected.

## consequences

- `wbhk upgrade` now cryptographically proves a new binary's GitHub-Actions build provenance before
  installing — verified end-to-end against the real release under **both** Node and the **bun-compiled
  binary** (positive + a tampered binary is rejected by the subject-digest check). The pure, security-critical
  pieces (pinned identity, subject-digest check, statement decode) are unit-tested; the sigstore orchestration
  + the shim are coverage-excluded io wiring, verified empirically.
- `install.sh` (`curl | sh`) stays checksum-only (it's shell — can't run sigstore); Homebrew checksum-pins;
  npm has its own provenance (`npm audit signatures`). The in-process check is `wbhk upgrade`'s.
- Adds ~1–2 MB (sigstore + the 7 KB embedded root) to the bundle. The trust root is pinned at build → refresh
  it when sigstore's root rotates (the `gen-trusted-root.mjs` script + a note).

## alternatives considered

- **Best-effort `gh attestation verify` / `cosign` shell-out.** Rejected (the founder chose real in-process) —
  it only verifies for users who already have those tools installed + authed.
- **Fetch the trust root over TUF at runtime.** Rejected — `bun --compile` drops the seeds; embedding is more
  robust (offline, no per-run network) at the cost of refreshing on rotation.
- **Relax the tlog/SCT thresholds to dodge the bun ECDSA failure.** Rejected — it weakens verification (drops
  transparency-log proof) and the verifier checks present material regardless of threshold anyway; the shim
  fixes the root cause and keeps full verification.
